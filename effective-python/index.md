- 为什么同样的功能，有些实现更加高效？

## 第 2 章 列表与字典

### 第 11 条 学会对序列做切片

`list`、`str`与`bytes`都可以获取原序列的某个子集（即切片）。凡是实现了`__getitem__`与`__setitem__`两个特殊方法的类都可以切割[第 43 条 自定义的容器类型应该从 collections.abc 继](#第 43 条 自定义的容器类型应该从 collections.abc 继承)。

切割时所用的下标可以越界，但直接访问列表却不行，会让程序抛出异常。

切割出来的列表是一份全新的列表。如果元素是**非引用类型**，则不会影响到原列表。

把切片放在赋值符号的左侧可以将原列表中这段范围内的元素用赋值符号右侧的元素替换掉，但可能会改变原列表的长度。

### 第 12 条 不要在切片里同时指定起止下标与步进

不要把起止下标与步进值同时写在切片里。如果必须指定步进，尽量采用正数，且把起止下标都留空。若必须同时使用步进值与起止下标，应该分成两次来写。

```python
x = ['a', 'b', 'c', 'd', 'e', 'f', 'g', 'h']

y = x[::2]

z = y[1:-1]
```

若程序没有那么多时间或内存分两步操作，可改用 itertools 模块中的 islice 方法[第36条 考虑用itertools拼装迭代器与生成器](#第36条 考虑用itertools拼装迭代器与生成器)，其起止位置与步进值不能是负数。

## 第 9 章 测试与调试

### 第 80 条 考虑用 pdb 做交互调试

Python 内置的`breakpint`函数可以用来触发调试器。该函数的效果与先引入内置的 pdb 模块然后运行`set_trace`函数效果是一样的。

一旦运行`breakpoint`函数，程序会在即将执行下一条语句的地方暂停，运行 Python 程序的终端窗口会进入交互调试界面。

![PDB交互调试界面](/images/202606/9/da0a2dadb95f8901c0eb4aa08e95d720_MD5.jpeg "PDB交互调试界面")

++事后调试（post-mortem debugging）++，当发现程序抛出异常并崩溃后，想通过调试器看到抛出异常的那一刻究竟是什么样的。大概率此时我们也不确定应该在哪里调用`breakpoint`函数。

此时可以运行`python -m pdb -c continue <program_path>`命令，会让 pdb 在启动受测程序之后立刻向前推进，直到遇到断点或出现异常为止。

还有一种方法触发事后调试机制，就是先在普通的 Python 解释器里执行受测代码，待遇到未被捕获的异常后，再引入 pdb 模块并调用其`pm`函数（`import pdb; pdb.pm()`）。

![出现未捕获异常后进入PDB](/images/202606/9/4b37e8df33feee46bd11aee07e750d6c_MD5.jpeg "出现未捕获异常后进入PDB")
